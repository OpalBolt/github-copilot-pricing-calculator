Understanding token types

The same total token count can cost very differently depending on how much of it is fresh context, reused context, or model output.

Input tokens
New prompt/context tokens sent on a fresh request — like instructions, code, or chat history the model has to read from scratch. They cost more than cached input because the model must process the full context again.
Cached tokens
Reused input from the same session, such as repeated system text or unchanged file context. Providers can reuse prior computation here, so cached input is much cheaper and often makes up most of a coding session's token mix.
Output tokens
Tokens the model generates in its reply. Generation is sequential and compute-heavy, so output usually has the highest per-token rate even when it is a small share of total tokens.

Table columns

Est. cost
The estimated USD cost for one run of your configured token mix. It combines your total tokens with the Input / Cached / Output percentages and each model's rates.
Runs
How many times you could repeat that exact token mix within your budget. This is Budget ÷ Est. cost, so larger budgets or cheaper models increase the count.
Tokens
The total raw token volume your budget can afford for that model. It is Runs × Total tokens, so it moves with Runs but tells you the overall token capacity.
Relative cost
A bar that shows cost relative to the most expensive model currently visible. The value is displayed in AI credits, where 1 credit equals $0.01.

That is why the Input / Cached / Output sliders matter so much for the Est. cost column: the same total can land at a very different price once each token type uses its own $/1M rate.
